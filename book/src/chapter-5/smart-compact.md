# 5.2 pi-smart-compact 原理：两阶段增强压缩

> Smart Compact 是 pi 内置 Compaction 机制的增强版——它不是简单地截断历史，而是"聪明地"决定保留什么、丢弃什么。

## 为什么需要增强压缩？

pi 内置的 Compaction 会在上下文接近上限时自动压缩旧对话，但它不够"聪明"：

```
内置 Compaction：
  压缩前 100 轮 → 压缩后生成一段通用摘要
  问题：摘要太粗糙，关键细节丢失
```

Smart Compact 的改进——两阶段压缩：

| 阶段 | 触发时机 | 做什么 |
|------|----------|--------|
| Phase 1 | 上下文使用率 > 60% | 提取关键信息：决策、约定、文件修改、结论 |
| Phase 2 | 上下文使用率 > 80% | 丢弃低价值信息：重复文件读取、中间调试输出、失败的尝试 |

```
第一阶段（60%）：
  80K 原始对话 → 40K 增强摘要（保留所有决策和关键结论）

第二阶段（80%）：
  40K 增强摘要 → 20K 精炼摘要（只保留当前任务所需的信息）
```

## 配置

### 安装

```json
{
  "packages": ["pi-smart-compact"]
}
```

### 命令

| 命令 | 用途 |
|------|------|
| `/smart-compact` | 手动触发压缩 |
| `/smart-compact-config` | 查看当前配置和模式 |

### 自动/手动模式

在 `.pi/settings.json` 中配置：

```json
{
  "smart-compact": {
    "auto": "auto"
  }
}
```

- `auto`（默认）：自动在合适的时机触发压缩
- `manual`：只在用户执行 `/smart-compact` 时触发

## 压缩保留什么？

Smart Compact 优先保留以下内容：

| 优先级 | 内容类型 | 为什么保留 |
|--------|----------|-----------|
| 🔴 最高 | 用户明确的要求和约束 | 这是任务目标 |
| 🟠 高 | 关键决策和选择理由 | 避免 AI 重复讨论已否决的方案 |
| 🟡 中 | 文件修改记录 | 让 AI 知道哪些文件已改过 |
| 🟢 低 | 工具调用过程（读取了什么、搜索了什么） | 可以重新执行 |
| ⚪ 最低 | 失败的尝试和调试过程 | 已经学到了教训 |

## 最佳实践

- **不要等 80% 才关注**：60% 时 AI 的质量就开始下降，Phase 1 是最佳的预防时机
- **手动触发适合关键操作前**：在开始一个重要的重构之前，手动 `/smart-compact` 清理上下文
- **配合 context-manager**：Smart Compact 压缩对话历史，Context Manager 的 distill 压缩工具输出，两者互补
