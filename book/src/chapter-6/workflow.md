# 6.3 pi-workflow 原理：子代理编排

> pi-workflow 是 pi-atelier 的"任务分发器"——它让 AI 把复杂任务拆成多个子代理并行执行，最后汇总结果。

## 为什么需要子代理？

单个 AI 会话的上下文窗口有限。当任务复杂到需要：
- **同时搜索多个信息源**
- **独立执行互不干扰的操作**
- **用不同的视角审查同一份代码**

单个 AI 只能串行处理，效率低，而且所有信息都挤在一个上下文里。

子代理解决了这个问题——每个子代理有独立的上下文窗口，执行完只把结果返回给主代理。

## 工作原理

```
主代理
  │ "调研 Bun 和 Node.js 的性能差异"
  │
  ├──→ 子代理 1（搜索专家）
  │     独立上下文窗口
  │     搜索网页资料
  │     返回关键数据点
  │
  ├──→ 子代理 2（源码专家）
  │     独立上下文窗口
  │     搜索 GitHub 仓库
  │     返回 benchmark 数据
  │
  └──→ 子代理 3（历史专家）
        独立上下文窗口
        搜索历史会话
        返回之前的讨论记录
  │
  ▼
主代理综合三个结果 → 输出最终建议
```

### 子代理的特性

| 特性 | 说明 |
|------|------|
| 独立上下文 | 每个子代理有自己的上下文窗口，不会污染主会话 |
| 独立工具集 | 可以限制子代理能用的工具 |
| 错误隔离 | 一个子代理出错不影响其他 |
| Token 效率 | 只有最终结果回到主上下文，中间过程不占主窗口 |

## 可用的子代理

子代理定义在 `~/.pi/agent/agents/*.md` 中，每个 `.md` 文件定义一个子代理的角色和工具集：

| 子代理 | 用途 | 工具 |
|--------|------|------|
| `pv-explorer` | 代码探索——分析架构、调用链、设计模式 | read, grep, find, ls |
| `pv-reviewer` | 独立方案审查——检查架构违反、依赖方向错误 | read, grep, find, ls |
| `pv-executor` | 执行代码变更——按方案实施、让测试通过 | 全部工具 |
| `pv-simplifier` | 代码简化——识别重复、低效写法 | read, grep, find, ls |
| `fo-analyzer` | 因子分析——运行分析脚本 + 解析结果 | bash, read |
| `fo-verifier` | 因子验证——运行最终回测 + 输出报告 | bash, read |
| `fr-searcher` | 因子搜索——搜索文献和源码 | read, grep, find, ls |
| `fr-writer` | 因子写作——综合研究结果 | read, grep, find, ls |
| `security-auditor` | 安全审查——检查安全漏洞 | read, grep, find, ls |

## 使用场景

### 场景 1：方案审查（Plan-Verify 流程）

在做复杂变更前，用 `pv-reviewer` 独立审查方案：

```
主代理：我设计了一个重构方案，让审查员看看有没有问题。
  │
  └──→ pv-reviewer
        "审查以下方案：将认证模块从 JWT 切换到 session-based"
        
        审查结果：
        ✅ 依赖方向正确
        ⚠️ 违反了数据模型不变量 #3（用户 session 应该是不可变的）
        ❌ 缺少测试覆盖——新 session 存储需要集成测试
```

**价值**：审查员从独立视角看方案，能发现主代理"当局者迷"的问题。

### 场景 2：代码探索

用 `pv-explorer` 结构化分析代码，不污染主上下文：

```
主代理：我需要理解这个模块的架构。
  │
  └──→ pv-explorer
        "分析 src/auth/ 目录的架构、调用链路、设计模式"
        
        返回：
        - 模块结构图
        - 核心函数调用链
        - 使用的设计模式（中间件链、策略模式）
        - 依赖方向
```

**价值**：探索过程可能读取几十个文件，如果都在主上下文里会占满空间。子代理探索完只返回精华。

### 场景 3：安全审查

用 `security-auditor` 检查代码中的安全漏洞：

```
主代理：这段代码处理用户输入，帮我做安全审查。
  │
  └──→ security-auditor
        "审查 src/api/handlers/ 的输入验证和注入风险"
        
        发现：
        ⚠️ SQL 注入风险：query 参数直接拼接 SQL
        ⚠️ XSS 风险：用户输入未转义就返回 HTML
        ✅ 认证检查：所有端点都有 auth middleware
```

## 子代理 vs 普通对话

| 维度 | 普通对话 | 子代理 |
|------|----------|--------|
| 上下文 | 所有信息在一个窗口 | 每个子代理独立窗口 |
| 并行 | 串行处理 | 可并行 |
| 出错影响 | 全局影响 | 隔离 |
| Token 消耗 | 中间过程全占主窗口 | 只有结果占主窗口 |
| 适用场景 | 简单任务、问答 | 复杂研究、多方审查 |

> 💡 **经验法则**：如果任务需要读取大量文件但最终只需要一个结论，用子代理。如果需要多轮交互讨论，用普通对话。

## 注意事项

- 子代理是**一次性的**——执行完返回结果就结束，不会保持状态
- 子代理的工具集是**受限的**——`pv-explorer` 只有只读工具，不能改文件
- 子代理看不到主会话的完整上下文——只看到你传给它的 `task` 描述
- 因此，给子代理的 task 描述要**充分具体**——包含所有必要背景信息
